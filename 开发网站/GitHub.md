# GitHub 完全使用指南

本指南将从入门到进阶，全面介绍 GitHub 的核心功能和日常使用方法，帮助你快速掌握这个强大的代码托管平台。

---

## 1. 引言与基础概念

### 1.1 Git 和 GitHub 的区别

* **Git**：本地版本控制系统，用于管理代码的历史版本，在你自己的电脑上运行。
* **GitHub**：基于 Git 的在线代码托管平台，提供协作功能、可视化界面和社区互动。

简单来说：Git 是工具，GitHub 是平台。

### 1.2 核心概念

| 概念 | 解释 |
|------|------|
| **Repository（仓库）** | 存放项目代码、文档和资源的地方 |
| **Branch（分支）** | 独立开发的代码线，可以与主分支并行 |
| **Commit（提交）** | 记录代码更改的快照 |
| **Pull Request（PR）** | 请求将你的代码合并到其他分支 |
| **Issue（问题）** | 跟踪任务、Bug 和功能需求 |
| **Fork** | 复制他人仓库到自己账号下 |
| **Clone** | 将远程仓库下载到本地 |

## 2. 账号注册与配置

### 2.1 创建 GitHub 账号

1. 访问 [GitHub 官网](https://github.com)
2. 点击 "Sign up" 创建新账号
3. 填写邮箱、用户名和密码
4. 完成验证和设置步骤

### 2.2 配置 Git（本地环境）

安装 Git 后，在命令行中设置你的身份信息：

```bash
git config --global user.name "你的用户名"
git config --global user.email "你的邮箱"
```

这样提交代码时，系统会知道是谁做的更改。

## 3. 核心功能详解

### 3.1 Repository（仓库）

**什么是仓库？**
- 仓库是 GitHub 上最基本的存储单位
- 相当于项目的容器，包含所有代码、文档和相关资源

**仓库类型：**
- **公开仓库（Public）**：任何人都可以查看和 Fork
- **私有仓库（Private）**：只有被邀请的用户才能访问

**主要组成部分：**
- **代码（Code）**：项目的源代码文件
- **Issues**：任务和问题跟踪系统
- **Pull Requests**：代码审查和合并请求
- **Actions**：自动化工作流
- **Wiki**：项目文档
- **Projects**：任务看板

**使用场景：**
- 个人项目开发和备份
- 团队协作开发
- 开源项目分享和社区贡献

### 3.2 Project（项目看板）

**什么是 Project？**
- GitHub 的任务管理工具，类似于 Trello 或 Jira
- 以看板形式组织任务，可视化工作流程

**Project 类型：**
- **仓库级 Project**：仅管理单个仓库的任务
- **组织级 Project**：可跨多个仓库管理任务

**主要功能：**
- **列（Columns）**：表示任务状态，如 "待办"、"进行中"、"已完成"
- **卡片（Cards）**：代表具体任务，可以关联 Issue、PR 或添加笔记
- **自动化（Automation）**：设置规则自动更新任务状态

**使用场景：**
- 敏捷开发中的 Sprint 规划
- 项目进度跟踪
- 团队工作分配和协作

### 3.3 Codespace（代码空间）

**什么是 Codespace？**
- GitHub 提供的云端开发环境
- 基于 VS Code，可直接在浏览器中编写和运行代码

**主要特点：**
- **即开即用**：无需在本地安装环境，打开浏览器即可开发
- **基于容器**：每个 Codespace 是独立的容器，包含完整开发工具
- **与 GitHub 集成**：可直接提交代码、创建 PR
- **自定义配置**：通过 `.devcontainer` 文件自定义开发环境

**使用场景：**
- 在不同设备间无缝切换开发
- 快速测试开源项目
- 统一团队开发环境，避免环境不一致问题
- 在线教学和代码演示

### 3.4 Organization（组织）

**什么是 Organization？**
- 团队或公司使用的 GitHub 账号类型
- 用于集中管理多个仓库和成员权限

**主要组成：**
- **成员（Members）**：组织内的用户，可分配不同权限
- **团队（Teams）**：成员分组，方便批量管理权限
- **仓库（Repositories）**：组织拥有的代码仓库
- **项目（Projects）**：跨仓库的任务看板

**使用场景：**
- 公司或团队协作开发
- 开源组织管理多个项目
- 需要精细权限控制的多仓库管理

### 3.5 Gist（代码片段）

**什么是 Gist？**
- GitHub 提供的轻量级代码分享工具
- 本质上是一个迷你 Git 仓库，支持版本控制

**主要特点：**
- **轻量级**：适合分享小段代码或配置
- **多文件支持**：可在一个 Gist 中包含多个文件
- **可见性控制**：支持公开（Public）和私密（Secret）两种模式
- **语法高亮**：自动识别编程语言并高亮显示
- **可嵌入**：可通过代码嵌入到博客或网站中

**使用场景：**
- 分享代码片段或示例
- 存储常用配置文件
- 记录学习笔记或小教程
- 在论坛或聊天中快速分享代码

**多文件 Gist 示例：**
```
main.py         # 主程序文件
requirements.txt # 依赖列表
README.md       # 使用说明
```

## 4. 日常操作指南

### 4.1 仓库操作

**创建新仓库：**
1. 点击右上角 `+` 图标，选择 "New repository"
2. 填写仓库名称、描述
3. 选择公开或私有
4. 可选择初始化 README、.gitignore 等文件
5. 点击 "Create repository"

**克隆仓库到本地：**
```bash
git clone https://github.com/用户名/仓库名.git
```

**推送本地代码到远程：**
```bash
git add .                      # 添加所有更改
git commit -m "提交说明"       # 提交更改
git push origin 分支名          # 推送到远程仓库
```

### 4.2 分支管理

**创建并切换分支：**
```bash
git checkout -b 新分支名
```

**切换到已有分支：**
```bash
git checkout 分支名
```

**推送分支到远程：**
```bash
git push origin 分支名
```

**删除本地分支：**
```bash
git branch -d 分支名
```

### 4.3 同步更新

**拉取远程更新：**
```bash
git pull origin 分支名
```

**或者分步操作：**
```bash
git fetch origin        # 获取远程更新信息
git merge origin/分支名  # 合并远程更新到本地
```

### 4.4 处理冲突

当多人修改同一文件时，可能会产生冲突：
1. Git 会在冲突文件中标记冲突部分
2. 手动编辑文件，解决冲突
3. 解决后，使用 `git add` 添加文件
4. 继续完成提交

## 5. 协作与项目管理

### 5.1 协作开发流程

**标准协作流程：**
1. **Fork**：复制他人仓库到自己账号
2. **Clone**：将 Fork 的仓库克隆到本地
3. **修改**：在本地进行代码更改
4. **推送**：将更改推送到自己的 Fork 仓库
5. **Pull Request**：向原仓库发起合并请求
6. **Review**：原仓库维护者审查代码
7. **合并**：审查通过后合并代码

### 5.2 Issue 管理

**创建 Issue：**
1. 进入仓库，点击 "Issues" 选项卡
2. 点击 "New issue"
3. 填写标题和描述
4. 可添加标签、指派人、里程碑
5. 点击 "Submit new issue"

**Issue 功能：**
- **标签（Labels）**：分类 Issue，如 bug、feature、question
- **指派人（Assignees）**：分配任务负责人
- **里程碑（Milestones）**：关联到特定版本或阶段
- **评论**：讨论任务细节

### 5.3 代码审查

**创建 Pull Request：**
1. 推送代码到远程分支后，GitHub 会提示创建 PR
2. 填写 PR 标题和描述
3. 选择要合并的分支
4. 点击 "Create pull request"

**代码审查流程：**
1. 审查者查看代码更改
2. 可对特定代码行添加评论
3. 可请求更改或批准合并
4. 审查通过后，维护者可合并 PR

## 6. 高级功能与技巧

### 6.1 GitHub Pages

**什么是 GitHub Pages？**
- GitHub 提供的免费静态网站托管服务
- 可用于展示项目文档、个人博客等

**使用方法：**
1. 在仓库设置中开启 GitHub Pages
2. 选择源分支（通常是 `main` 或 `gh-pages`）
3. 可选使用主题或自定义 HTML
4. 网站会自动部署到 `https://用户名.github.io/仓库名/`

### 6.2 GitHub Actions

**什么是 GitHub Actions？**
- GitHub 的持续集成/持续部署（CI/CD）工具
- 可自动化测试、构建、部署等流程

**配置方法：**
1. 在仓库中创建 `.github/workflows/` 目录
2. 添加 YAML 配置文件定义工作流
3. 每次推送或 PR 时自动触发

### 6.3 标签管理

**创建标签：**
```bash
git tag v1.0       # 创建轻量标签
git tag -a v1.0 -m "版本1.0"  # 创建带注释的标签
```

**推送标签到远程：**
```bash
git push origin v1.0    # 推送单个标签
git push origin --tags  # 推送所有标签
```

### 6.4 实用技巧

**快捷键：**
- `t`：快速搜索文件
- `y`：锁定当前文件版本的 URL
- `s`：搜索当前仓库
- `l`：查看文件行号

**其他技巧：**
- **Star**：收藏感兴趣的项目
- **Watch**：关注项目动态，接收通知
- **Notifications**：统一管理所有通知
- **模板**：为 Issue 和 PR 创建模板，标准化提交格式

## 7. GitHub 生态系统关系

### 7.1 核心功能关系图

```
Organization (组织/团队)
 ├── Teams (团队成员分组)
 ├── Projects (跨仓库任务看板)
 ├── Repositories (代码仓库)
 │    ├── Code / Issues / Pull Requests
 │    ├── Wiki / Actions / Pages
 │    └── 可打开为 Codespace (在线开发)
 └── Members (成员管理与权限控制)

Gist (代码片段) - 独立的轻量级代码分享工具
```

### 7.2 功能对比表

| 功能 | 主要用途 | 是否支持多人协作 | 是否适合完整项目 |
|------|----------|------------------|------------------|
| Repository | 完整项目代码托管 | ✅ | ✅ |
| Project | 任务管理与看板 | ✅ | ❌ |
| Codespace | 云端开发环境 | ✅ | ✅ |
| Organization | 团队与权限管理 | ✅ | ✅ |
| Gist | 代码片段与小文件分享 | 🚫 (主要单人使用) | ❌ |
